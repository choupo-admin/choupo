The black box is the problem

A student opens the big commercial simulator, picks a thermodynamic method from a list, clicks Run, and gets a number. Which equation? Which assumption? Why that K-value? Hidden. You are taught to trust the box.

The black box: trust the number

A method is a name in a dropdown. The maths is sealed. When the answer is wrong you cannot see where — only that it differs from the lab.

Choupo: see the maths

The model is open in the source and it narrates itself as it solves. You choose a model because you saw it fit the data — not because a badge said "recommended".

What it does

A real simulator — steady-state, dynamic, batch — and a property bench that lets you build and trust the thermodynamics before you wire a flowsheet.

Flowsheets

Flash, distillation, reactors (CSTR/PFR/Gibbs), heat exchange, rotating equipment, recycle with Newton-on-tears. Four solvers, one per problem class.

Property Explorer

Navigate any component or mixture and plot its properties interactively — vapour pressure, density, compressibility — and see the curves before you commit. Runs in the browser; nothing to install.

The property bench

Create a component from its groups, overlay models on your measured data, test the data's consistency, fit parameters — then carry a curated file into the run.

Membranes (NF/RO)

A spiral-wound module with solution-diffusion + film model — the research edge a general simulator does not reach.

File-first cases

Plain-text dictionaries on disk are the source of truth. Edit them in any editor — increasingly with an AI agent at your side.

Sensitivity & optimisation

Sweep a parameter, minimise a cost, fit to data — the outer-driver layer wraps every run.

Sizing & costing

Guthrie module costs, utility allocation by temperature level — the report a project actually needs.

Install & build it locally

Want the full simulator on your own machine — all four binaries, every tutorial, the regression suite? It builds from source with zero external dependencies: just a C++17 compiler (g++) and GNU make. No Boost, no Eigen, no CMake. Pick your OS.

Linux

The native home of Choupo — the smoothest path.

1. Install the toolchain (g++ + GNU make + git):
# Debian / Ubuntu
sudo apt update && sudo apt install g++ make git

# Fedora / RHEL
sudo dnf install gcc-c++ make git
2. Clone and build the native release:
git clone https://github.com/choupo-admin/choupo.git
cd choupo
make all                # native release (-O2) -> choupoSolve, choupoBatch, choupoCtrl, choupoProps
3. Source the shell helpers, then run a tutorial:
source etc/bashrc       # adds bin/ + project root to PATH (runCase, listCases, ...)
runCase tutorials/steady/flash/flash01_benzene_toluene
4. Run the full regression suite to confirm a healthy build:
bin/runTests
...
PASS 117 / FAIL 0   <- your success signal: tutorials solve + golden-master KPIs match, zero failures
Optional — the browser GUI: (Node.js required)
cd gui
npm install
npm run dev             # Vite dev server on http://localhost:5173
Two traps to know. Use make all, not a bare make — bare make resolves to the WebAssembly target (a known include-order bug). And the helpers (runCase, listCases) only appear on your PATH after source etc/bashrc in that shell; re-source it in each new terminal, or call ./choupoSolve <case> directly.

Windows

Two honest routes. WSL2 is the recommended one — it is a real Linux, so the build is identical and trouble-free.

Route A — WSL2 (recommended). In PowerShell as Administrator:
wsl --install            # installs Ubuntu under WSL2; reboot when asked
Then open the Ubuntu terminal and follow the Linux steps above verbatim:
sudo apt update && sudo apt install g++ make git
git clone https://github.com/choupo-admin/choupo.git
cd choupo
make all
source etc/bashrc
runCase tutorials/steady/flash/flash01_benzene_toluene
bin/runTests             # -> PASS .. / FAIL 0
Route B — MSYS2 / MinGW-w64 (native, no Linux layer). Install MSYS2, open the MSYS2 MINGW64 shell, then:
pacman -S mingw-w64-x86_64-gcc make git
git clone https://github.com/choupo-admin/choupo.git
cd choupo
make all
source etc/bashrc
runCase tutorials/steady/flash/flash01_benzene_toluene
bin/runTests
Which is smoother? WSL2 — by a clear margin. It runs the exact Linux toolchain, so paths, the source etc/bashrc helpers, and bin/runTests behave precisely as documented. MSYS2 works (the code is portable C++17 with no OS-specific deps), but you must stay in the MINGW64 shell and use its g++/make. Same trap as Linux: make all, never a bare make (that builds WebAssembly), and the helpers need source etc/bashrc first.

macOS

Choupo's Makefile invokes g++. On a Mac that name usually points at Apple's clang, which compiles the C++17 just fine — or you can install the real GNU GCC with Homebrew.

1a — simplest: Xcode command-line tools (gives clang as g++) + git:
xcode-select --install   # installs clang (aliased as g++) + make + git
1b — or GNU GCC via Homebrew (installs g++-NN + GNU make):
brew install gcc make git
2. Clone and build:
git clone https://github.com/choupo-admin/choupo.git
cd choupo
make all                 # clang as g++ builds C++17 cleanly; or pass CXX=g++-14 to force GNU GCC
3. Source the helpers and run a tutorial:
source etc/bashrc
runCase tutorials/steady/flash/flash01_benzene_toluene
4. Confirm with the regression suite:
bin/runTests             # -> PASS .. / FAIL 0
clang vs g++. Apple ships clang++ under the name g++; it builds Choupo's standard C++17 with no changes. If you installed Homebrew GCC and want the genuine GNU compiler, point the build at it with make all CXX=g++-14 (match your installed version, e.g. g++-13). Same two traps as everywhere: make all not a bare make (which builds WebAssembly), and source etc/bashrc before the runCase helper works.

Prefer not to install anything? The browser app runs the same C++ solver as WebAssembly — see Run it now above. Full build notes live in the README.
